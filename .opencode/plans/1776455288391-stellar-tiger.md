# Add Upstash Rate Limits To Survey API

## Goal

Protect the anonymous survey endpoints from abuse with Upstash Ratelimit, while keeping normal survey usage smooth. Apply route-level limits to `unlock`, `save`, and `submit`. Fail closed when Upstash is not configured.

## Recommended Approach

1. Add Upstash dependencies and env vars.
2. Create one shared server-side rate limit helper that follows the repo's existing singleton utility pattern.
3. Apply route-level checks inside the three POST handlers before expensive work.
4. Return consistent JSON `429` responses with retry metadata.

## Files To Modify

- `package.json`
- `.env.example`
- `src/env.ts`
- `src/lib/ratelimit.ts` (new)
- `src/app/api/survey/unlock/route.ts`
- `src/app/api/survey/save/route.ts`
- `src/app/api/survey/submit/route.ts`

## Implementation Details

### 1. Dependencies

Add:

- `@upstash/ratelimit`
- `@upstash/redis`

### 2. Environment Contract

Update `src/env.ts` and `.env.example` with:

- `UPSTASH_REDIS_REST_URL`
- `UPSTASH_REDIS_REST_TOKEN`

Make both required in `src/env.ts` so the existing `hasRequiredSurveyEnv()` guard fails closed when Upstash is missing.

Update the survey API fallback message so it mentions the Upstash env vars in addition to the existing survey env vars.

### 3. Shared Helper

Create `src/lib/ratelimit.ts`.

Follow the same shape as `src/db/client.ts`:

- lazily create a shared Upstash Redis client
- lazily create cached `Ratelimit` instances
- keep the public API small and route-focused

Planned helper responsibilities:

- derive a stable actor key from the current request
- run the correct limiter for a route
- return enough data for a `429` JSON response and `Retry-After` header

### 4. Key Strategy

Use hashed / privacy-preserving identifiers already consistent with the app:

- `unlock`: fingerprint only via `createFingerprintHash(headers)`
- `save`: `hashSessionToken(sessionToken)` when available, otherwise fingerprint, plus `responseId`
- `submit`: same strategy as `save`, plus `responseId`

Proposed key shapes:

- `survey:unlock:${fingerprintHash}`
- `survey:save:${actorKey}:${responseId}`
- `survey:submit:${actorKey}:${responseId}`

This avoids raw session tokens and fits the app's anonymous-session model.

### 5. Initial Limit Policy

Keep the first pass simple with constants in `src/lib/ratelimit.ts` rather than more env/config.

Recommended initial limits:

- `unlock`: `10 requests / 10 minutes`
- `save`: `120 requests / 5 minutes`
- `submit`: `5 requests / 10 minutes`

Use Upstash sliding windows for all three.

This matches the requested posture:

- lenient for autosave
- stricter for access-code brute force
- strictest for repeated submit attempts

### 6. Route Integration Points

#### `src/app/api/survey/unlock/route.ts`

Add the rate limit check:

- after `request.json()` parsing
- before access-code verification
- before any DB reads/writes

If limited, return `429` immediately.

#### `src/app/api/survey/save/route.ts`

Add the rate limit check:

- after `request.json()` parsing
- after cookie extraction
- before `authorizeResponseAccess()` and before save work

Use parsed `responseId` as part of the key.

#### `src/app/api/survey/submit/route.ts`

Add the rate limit check in the same location pattern as `save`.

### 7. Response Behavior

For rate-limited requests:

- return `429`
- JSON body should include at least:
  - `message`
  - `retryAfterSeconds`
- set `Retry-After` response header

If straightforward from the Upstash result, also include:

- `X-RateLimit-Limit`
- `X-RateLimit-Remaining`
- `X-RateLimit-Reset`

Keep the response style aligned with the current `jsonError(...)` usage in each route.
