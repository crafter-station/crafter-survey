# Community Survey Plan

## Goal

Replace the starter page with a dark, editorial multi-section survey experience that:

- requires the access code `JUST-SHIP-IT`
- matches the visual language of `cuevaio/site`
- persists all progress in Neon via Drizzle
- autosaves with debounce plus section checkpoints
- resumes anonymous users where they left off
- keeps survey structure DB-driven so questions/options/copy can be edited in production through the DB visualizer

## Follow-up Refinement

Apply a small visual/content refinement pass on top of the implemented survey:

- replace `Anthony & Railly` with `the Crafter Station team`
- remove the current purple-tinted glow/focus treatment and keep the palette neutral
- add system-driven dark/light theme support, without adding a user-facing toggle
- strengthen the grain treatment so it feels closer to `cuevaio/site`, without introducing the full shader background

## Key Decisions

- Keep the survey on `/` as a single App Router page.
- Use one main client component for interactive state and Framer Motion transitions.
- Use route handlers for unlock/save/submit instead of server actions.
- Use a versioned, row-based survey schema rather than question-specific DB columns.
- Use a secure anonymous cookie as the primary session id and a salted fingerprint hash as server-side recovery fallback.
- Treat the final optional `name`, `email`, `phone`, and extra note fields as normal survey questions in the last section.
- Prefer Framer Motion only for section-level transitions and progress state changes, not heavy per-field animation.

## Current Repo Constraints

- Survey copy is DB-backed, so changing `src/db/seed/community-survey.ts` alone will not update an already-seeded active survey.
- The current purple tint comes from:
  - `src/app/globals.css` body radial gradient
  - `src/components/survey/survey-shell.tsx` top radial glow
- The app is currently dark-only:
  - `src/app/layout.tsx` uses `colorScheme: "dark"` and a dark-only `themeColor`
  - `src/app/globals.css` hardcodes dark tokens in `:root`
  - multiple survey components use direct `text-white`, `bg-white`, `bg-black`, and `border-white/*` classes
- Grain already exists, but only as a single inline noise layer in `src/components/survey/survey-shell.tsx`

## Recommended Architecture

### Page / Server Boundary

- Keep `src/app/page.tsx` as a Server Component.
- On load, read cookies and request headers.
- Resolve the current anonymous session in this order:
  1. session cookie token
  2. fingerprint hash fallback
- Load either:
  - the user’s in-progress response and its survey version, or
  - the active survey version with no answers if no draft exists
- Pass a serialized survey payload into a client component such as `SurveyExperience`.

### Client Boundary

- `SurveyExperience` owns:
  - section index / current section
  - local answer state
  - dirty tracking
  - debounced autosave
  - immediate checkpoint saves on section changes
  - Framer Motion transitions
  - submit flow and completion state
- Question rendering stays data-driven from DB-loaded definitions.

### Server Mutations

- Add route handlers:
  - `POST /api/survey/unlock`
  - `POST /api/survey/save`
  - `POST /api/survey/submit`
- Route handlers validate payloads, resolve the session, and persist changes in transactions.

- Intro gate screen for the access code.
- Progress header with section title, count, and save state.

## Data Model

Design the schema so survey content is stored as rows, not code constants.

### Tables

#### `surveys`

- long-lived survey identity
- fields: `id`, `slug`, `title`, `description`, `gate_code_hash`, timestamps

#### `survey_versions`

- frozen/publishable versions of a survey definition
- fields: `id`, `survey_id`, `version_number`, `status`, `title`, `description`, `completion_title`, `completion_description`, timestamps

#### `survey_sections`

- ordered sections within a version
- fields: `id`, `survey_version_id`, `key`, `title`, `description`, `sort_order`, timestamps

#### `survey_questions`

- dynamic questions editable in the DB visualizer
- fields: `id`, `survey_section_id`, `key`, `prompt`, `help_text`, `question_type`, `required`, `placeholder`, `sort_order`, `validation_json`, `ui_json`, timestamps

Use plain text values for `question_type` instead of Postgres enums so new types can be introduced without migrations.

Initial supported types:

- `short_text`
- `long_text`
- `single_select`
- `multi_select`
- `email`
- `phone`

#### `survey_question_options`

- options for single- and multi-select questions
- fields: `id`, `question_id`, `key`, `label`, `help_text`, `sort_order`, `meta_json`

#### `anonymous_sessions`

- anonymous respondent identity
- fields: `id`, `session_token_hash`, `fingerprint_hash`, `unlocked_at`, `last_seen_at`, timestamps

#### `survey_responses`

- a single draft/submission for one session on one survey version
- fields: `id`, `session_id`, `survey_version_id`, `status`, `current_section_id`, `started_at`, `last_saved_at`, `submitted_at`

#### `survey_answers`

- generic answer storage
- fields: `id`, `response_id`, `question_id`, `question_key_snapshot`, `value_text`, `value_json`, `client_updated_at`, timestamps

### Why This Schema

- Adding or editing question copy/order/options is a row update, not a migration.
- Existing drafts stay tied to the exact version they started with.
- The user can safely manage content in the DB visualizer.
- The final contact fields fit naturally without special-case tables.

## Survey Versioning Workflow

Support editing in production through the DB, but structure it so drafts do not break.

Recommended operational model:

1. Keep one `active` survey version for new users.
2. Existing drafts always resume against their own `survey_version_id`.
3. For substantial survey changes, create a new version by copying rows, edit it in the DB visualizer, then mark it `active`.

Directly editing the active version should still work, but the safer default is new version activation.

## Anonymous Session Strategy

### Primary Identifier

- After a correct access code, issue an `HttpOnly`, `Secure`, `SameSite=Lax` cookie.
- Store only a hash of the raw session token in the DB.

### Fallback Identifier

- Build a salted fingerprint hash from request-derived values such as:
  - IP address / forwarded IP
  - `user-agent`
  - `accept-language`
  - optional platform hints
- Never store raw IP or browser strings.
- Use the fallback only to recover a draft when the cookie is missing.

### Resume Rules

- If a draft exists, open the last saved section and rehydrate answers.
- If the latest response is already submitted, show the completion screen instead of reopening the form.
- If fingerprint recovery finds a draft, reissue the session cookie.

## Save Strategy

Use a hybrid model.

### Debounced Autosave

- Text fields: save after a short debounce window.
- Select / checkbox interactions: save quickly because they are discrete.
- Only send changed answers.

### Immediate Checkpoints

Flush immediately when the user:

- advances to the next section
- goes back to a previous section
- reaches the final step
- submits
- hides or leaves the page using `keepalive`/`sendBeacon` style fallback if needed

### Server Behavior

- Upsert answers in a transaction.
- Update `current_section_id` and `last_saved_at` on each successful checkpoint.
- Reject stale writes using `client_updated_at` if two saves race.

## Main Risks

- Editing the active version directly in the DB can surprise in-flight drafts.
- Fingerprint recovery is imperfect and should remain a fallback, not the primary identity.
- Autosave increases write volume, so payloads should be delta-only.
- Dynamic `question_type` values increase flexibility but require loader validation.
- Overusing motion will make the form feel tiring; keep transitions scoped to sections and state changes.

## Recommended Default Tradeoffs

- Prefer resilience over cleverness.
- Prefer DB-driven survey content over hardcoded arrays.
- Prefer a small number of strong components over a large component tree.
- Prefer a calm site-matched aesthetic over high-energy “form app” UI.
