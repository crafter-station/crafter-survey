# Survey Conversational Refactor Plan

## Scope
- Show the large hero only before unlock.
- After unlock, use compact chrome and maximize usable space for chat/form.
- Make the chat/form toggle smaller and reduce post-unlock padding/gaps.
- Persist the full chat transcript server-side per survey response.
- Remove custom chat form UI entirely.
- Make all survey questions optional.
- Let users jump between sections freely without blocking.
- Let chat interpret free-form user messages, save multiple answers from one turn when possible, and keep form mode synced.
- Do not enumerate select options to the user unless clarification is needed.
- For select questions, let the LLM choose canonical option key(s) from the question options already present in context; if nothing is clearly mappable, use `other` only where that option already exists, otherwise ask a clarification.
- Make the chat feel like a real interview, not a field-by-field questionnaire.
- Use a hand-authored conversation playbook with broader topic-level turns, especially for the opening context section.

## Constraints
- Keep using the Vercel AI SDK + AI Gateway stack already in the repo.
- Keep `currentSectionId` as the form resume anchor / current focus anchor.
- Avoid extra derived-state effects; prefer server-authoritative hydration and event-driven refresh.
- Do not introduce OpenUI in this refactor; the repo does not use it today and this work is state/layout/tooling, not generative UI rendering.
- No backward-compatibility work is needed for older draft responses with required fields; assume the DB will be cleaned/reset.

## Recommended Approach
1. Split pre-unlock hero chrome from unlocked compact chrome.
- Refactor `src/components/survey/survey-shell.tsx` into a neutral page frame only.
- Use a dedicated pre-unlock hero component for `gate`, `missing`, and `unconfigured`.
- Use a compact unlocked chrome component for `survey` and `submitted`.
- Compact chrome should contain only the brand row, theme toggle, and the smaller mode toggle.
- Do not render the large survey title/description after unlock.

2. Reclaim space for the unlocked app surface.
- Reduce outer top padding and duplicated chrome after unlock.
- Make chat/form content fill the available shell height instead of sitting below a large header.
- Tighten `SurveyModeToggle`, `ProgressNav`, and post-unlock panel spacing before touching deeper component internals.
- Keep one clear context/progress signal per mode, not two.

3. Remove required-field gating across data, form flow, and submit.
- Update the active survey definition so no questions are marked `required: true`.
- Remove required badges/messages from the form UI.
- Remove section-level blocking in form navigation and allow submit without completeness checks.
- Keep answer-shape validation (email, phone, max length, max selections, `otherText` requirements) even though fields are optional.
- Replace the strictly sequential form footer behavior with navigation that can set `currentSectionId` directly without enforcing completeness.

4. Persist chat history server-side per response.
- Add a new DB table for chat state, keyed by `responseId`, instead of storing transcript in `survey_responses`.
- Store:
  - `messagesJson` for the structured transcript
  - `metaJson` for chat-only state such as `nextQuestionId` and `skippedQuestionIds`
  - timestamps and schema version
- Load transcript + chat meta in:
  - initial page load
  - `/api/survey/unlock`
  - `/api/survey/state`
- Make the server transcript authoritative in `/api/survey/chat`.
- Persist the user turn before generation, then persist the final assistant turn and updated meta on finish.
- Remove the client-side bootstrap message effect and seed the first assistant opener server-side when a response gets its first chat state.

5. Add a hand-authored conversation playbook layer above the survey schema.
- Add a new module such as `src/lib/survey/conversation-playbook.ts`.
- Model the chat as conversational clusters instead of one raw question at a time.
- Start with an explicit opening cluster for the first section:
  - `location`: fills `city` + `country`
  - `professional_snapshot`: fills `role` + `current_focus` and captures broader background context from the same answer
  - `experience_followup`: fills `experience_years` only if still unresolved after the broader snapshot
- Each cluster should define:
  - intent
  - covered question IDs
  - preferred opener
  - preferred follow-ups
  - extraction hints for likely multi-field answers
  - exit conditions for when the cluster is “good enough” to move on
- Keep canonical select-key validation exactly as planned; the playbook changes how we ask, not how answers are validated.

6. Replace the current tool/UI contract with a natural, non-linear conversation model.
- Remove `ask_question` entirely.
- Delete `src/components/survey/chat-question-picker.tsx` and all picker-based chat rendering.
- Replace single-answer saving with one turn-level tool such as `commit_survey_turn`.
- `commit_survey_turn` should accept:
  - `answers[]` with one or more question payloads
  - `skippedQuestionIds[]`
- For select payloads, use canonical option key fields instead of free-text matching fields:
  - `choiceKey` for single select
  - `choiceKeys[]` for multi select
  - `otherText` only when the chosen option supports free text
- Inside that tool:
  - validate all payloads with the existing answer pipeline
  - save multiple answers in one operation
  - merge skipped questions into chat meta
  - advance the active conversational cluster / unresolved cluster state rather than only one exact next field
  - recompute `currentSectionId` as soft conversational focus, not required-order state
  - return updated progress/chat state
- Update the chat prompt so the assistant:
  - extracts all confident answers from a free-form message
  - saves multiple answers when possible
  - asks only one follow-up question at the end of each turn
  - does not list available choices unless clarification is needed
  - uses only canonical option key(s) that already exist in the question context
  - does not paraphrase or invent option keys
  - uses `other` + `otherText` only when that option exists for the question
  - asks for clarification when matching is ambiguous or when no `other` fallback exists
  - prefers broad, human-sounding topic questions over literal field wording
  - never references UI controls or assumes a picker exists

7. Keep server validation authoritative for canonical option keys.
- Do not add or rely on aliases/synonyms in survey option metadata for chat matching.
- Remove any server-side option-matching helper used only to infer keys from free text.
- Keep validation strict in `prepareAnswerChanges()`:
  - invalid keys are rejected
  - duplicate multi-select keys are deduped
  - `otherText` is required when an `other`-style option needs it
  - `otherText` is ignored or rejected for non-free-text options as appropriate
- If the model is unsure which key applies, it should clarify before calling the tool rather than relying on backend heuristics.

8. Represent chat state as conversational goals instead of a single next field.
- Replace `SurveyChatMeta.nextQuestionId` with cluster-oriented state, for example:
  - `activeClusterKey`
  - `clusterStates`
  - `lastCompletedClusterKey`
  - `skippedQuestionIds`
- Update `src/lib/survey/chat-flow.ts` to compute:
  - the active cluster
  - unresolved questions inside that cluster
  - the best next follow-up from the playbook
  - a natural resume opener on reload
- A single user answer should be able to complete multiple raw questions inside the same cluster, e.g. `Lima, Perú` completing both `city` and `country`.

9. Keep form mode synchronized and resumable.
- Continue treating DB answers as the source of truth for the actual survey.
- Add a shared helper to compute the next conversational target from:
  - saved answers
  - active cluster / unresolved cluster goals
  - skipped question IDs
- Update `currentSectionId` from the current conversational focus so form mode reopens a sensible section, but do not treat it as a required-order cursor anymore.
- Keep the current form->chat dirty-flush behavior.
- On chat->form and form->chat switches, refresh from server-authoritative response + transcript state.
- Do not try to rewrite old transcript messages when the form edits existing answers; transcript remains history, answers remain truth.

10. Shape the first 5-8 turns explicitly.
- Opening:
  - `Hola. Voy a ayudarte a completar la encuesta. ¿Desde dónde nos escribes?`
- If the user says `Lima, Perú`, commit both `city` and `country` in one turn.
- Next broad follow-up:
  - `Buenísimo. Cuéntame un poco sobre ti: ¿a qué te dedicas y qué estás haciendo ahora?`
- If needed, a natural follow-up for focus:
  - `¿Y hoy estás más metido en trabajar para una empresa, freelancear, aprender, o construir algo tuyo?`
- Ask experience only if still unresolved after the broader professional snapshot.
- Move to interests with a broad transition like:
  - `¿Qué temas te tienen más metido ahora?`
- Keep all early turns interview-like, not label-like.

11. Update the client chat view to hydrate from persisted transcript.
- Initialize `useChat` from the server-provided transcript for the active `response.id`.
- Remove all active-question UI logic tied to tool parts for custom widgets.
- Keep chat rendering text-first and transcript-based.
- Refresh authoritative survey state after successful committed turns so form answers stay in sync.

  13. Give a malformed/nonexistent key in a mocked tool payload and confirm the server rejects it through validation.
  14. Confirm form navigation no longer blocks on unanswered questions and users can jump between sections freely.
  15. Reload after several chat turns and confirm the transcript resumes naturally at the current cluster rather than a raw field label.

  18. Submit a partially filled survey and confirm submitted state still works with compact chrome.

## Key Risks
- Persisted transcript shape is coupled to the app’s supported AI SDK message format, so validate stored payloads on read.
- Chat transcript and survey answers can drift if saved separately; keep server writes transactional where practical.
- Over-aggressive extraction can save wrong select answers; the prompt must force clarification on low-confidence mappings before the tool call.
- Some select questions do not support `other`; unmatched answers on those questions must clarify instead of auto-saving.
- Without a playbook layer, the model will keep drifting back toward field-by-field prompts even with better copy.
- Existing responses will not have a chat row; support lazy creation on first unlock/state/chat access.
